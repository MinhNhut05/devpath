## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, compact badge spacing |
| sm | 8px | Dense stacked text, chip spacing, micro-layout gaps |
| md | 16px | Default element spacing, card interior spacing, form row gaps |
| lg | 24px | Section padding inside cards and page blocks |
| xl | 32px | Layout gaps between primary regions and dashboard blocks |
| 2xl | 48px | Major section breaks, full empty-state vertical rhythm |
| 3xl | 64px | Page-level top/bottom separation on large screens |

Exceptions: 44px minimum touch target height for icon-only controls and compact sidebar/mobile nav triggers; 40px button height for standard actions already aligned with current UI; 256px/64px sidebar widths preserved from existing layout contract.

### Spacing Rules
- Use `md` (16px) as the default internal spacing for cards, form groups, and list items.
- Use `lg` (24px) for card padding in dashboard, explore, lesson side panels, and retry/empty states.
- Use `xl` (32px) between major page sections on dashboard and settings.
- Use `sm` (8px) between label and help text, icon and text, or stacked metadata.
- Avoid arbitrary spacing values except the preserved existing layout widths already in code (`64px`, `256px`) and touch-target exceptions.

### Source Notes
- Base 8-point scale: default selected because it matches current Tailwind usage and phase guidance.
- Existing custom spacing extensions `18` and `88` exist in Tailwind, but they are not part of the primary Phase 03 contract unless already required by legacy layout.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 600 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

### Typography Rules
- Declare exactly two weights for this phase: `400` and `600`.
- Use 16px body text for all primary reading content, descriptions, toast bodies, lesson metadata, and settings copy.
- Use 14px labels for buttons, input labels, navigation labels, chips, helper text, and compact table/list metadata.
- Use 20px heading size for card titles, section titles, empty-state headings, and error-state headings.
- Use 28px display size only for high-emphasis page greetings or major summary headers; do not use it inside sidebars or dense cards.
- Use Vietnamese diacritics consistently; never reduce readability with all-caps body copy.
- Keep sidebar/nav text at label size even when visually prominent.

### Mapping to existing tokens
- Prefer existing Tailwind aliases where they fit: `text-body`, `text-body-sm`, `text-h4`, and `text-h3` can be normalized during implementation, but Phase 03 contract should visually resolve to the four sizes above.
- If an existing token is visually larger than the contract role, the contract wins for newly touched Phase 03 screens.

### Source Notes
- Font family sourced from current Tailwind config and global CSS.
- Role reduction to 4 sizes and 2 weights is a deliberate Phase 03 simplification to stabilize UX rather than expand visual variety.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0A0A0F` (`dp-base`) | App background, main shell, protected-route loading shell, page backdrop |
| Secondary (30%) | `#161622` (`dp-surface`) | Cards, sidebar surfaces, lesson sidebar, elevated panels, empty/error containers |
| Accent (10%) | `#3B82F6` (`dp-primary`) | Primary CTA, active nav state, focus ring, progress highlight, skeleton emphasis edge, actionable links |
| Destructive | `#EF4444` (`dp-error`) | Destructive actions only, destructive icon accents, destructive confirmation emphasis |

Accent reserved for: primary CTA buttons, active navigation indicator, focused input ring, progress bar fill, current lesson highlight, inline actionable links, and toast emphasis for session/offline/error feedback. Do not use accent for every icon, every border, or all interactive surfaces.

### Color Distribution Rules
- 60% dominant color must remain the dark app shell and overall background field.
- 30% secondary color must define cards, sidebar, elevated containers, lesson outline panels, and empty/error-state cards.
- 10% accent is reserved and scarce. If a surface is not the primary action or active/current state, do not accent it.
- Use destructive red only for destructive actions or destructive warnings. Session expiration toast may use error styling, but must still remain non-blocking.
- Success, warning, and info colors may appear for semantic feedback, but they do not replace the dominant/secondary/accent ratio.

### Semantic Usage Rules
- Offline toast: warning/info styling with Vietnamese copy, not a blocking banner.
- Session-expired toast: destructive/error styling with redirect timer behavior.
- Empty states: use muted illustration tones with one accent CTA only.
- Skeletons: use low-contrast secondary overlays; no bright accent shimmer.

### Source Notes
- Colors sourced from existing Tailwind config and root CSS tokens.
- Accent restriction list derived from Phase 03 need for clarity during session/bootstrap and data-consistency work.

## Interaction Contract

### 1. Session Restore Boot Experience
- While `/auth/refresh` is in progress, render the existing protected shell structure instead of redirecting or showing a blank screen.
- Keep sidebar visible on routes that normally show `AppLayout`.
- Content area must show skeleton blocks, not spinner glyphs.
- Redirect decisions must wait until auth bootstrap finishes.
- Each tab performs its own refresh attempt on load; no cross-tab sync UI is required.
- Refresh retry count: exactly 0 retries after the initial attempt fails.

### 2. Protected/Admin Guard Behavior
- Guards must support three states: `loading`, `authenticated`, `unauthenticated`.
- `loading` state renders shell + skeleton.
- `unauthenticated` state redirects to `/login`.
- `authenticated` renders child route.
- `AdminGuard` must wait for bootstrap completion before evaluating `user.role`.

### 3. Loading Pattern
- Page-level loading uses skeleton shimmer only.
- Replace centered page spinners and full-screen `Đang tải...` blocks on touched Phase 03 screens.
- Skeleton shape should approximate final content layout: heading line, metadata line, primary card block, and list rows where relevant.
- For lesson sidebar, use 5-7 stacked row skeletons with one small heading skeleton.
- For dashboard, preserve existing card-grid silhouette.

### 4. Error Pattern
- Non-auth API errors surface as bottom-right toast and keep the page interactive.
- If the page has no previously loaded content and the request fails on first load, render a compact error card inside page content with retry button.
- Error card uses secondary surface, 20px heading, 16px body, one primary retry button.
- Do not use full-width blocking banners for transient API errors.

### 5. Offline Pattern
- On browser offline event or offline-detected query failure, show toast: `Không có kết nối mạng`.
- Requests may recover automatically when connection returns using React Query reconnect behavior.
- Do not show modal or full-page offline takeover in this phase.

### 6. Empty State Pattern
- Layout: centered card within content region, small illustration zone above heading, body text below, single primary CTA below body.
- Vertical order: illustration -> heading -> body -> CTA.
- CTA routes learner to `/explore` when no enrolled path exists.
- Empty states must feel encouraging, not punitive.

### 7. Navigation and Active State Pattern
- Active nav item uses accent highlight and stronger text contrast.
- Inactive nav items use muted text until hover/focus.
- Current lesson item in lesson sidebar uses accent/current-state emphasis without overwhelming the list.
- Optional track badge must be localized to Vietnamese and visually secondary.

### 8. Motion Pattern
- Use subtle transitions only: 150ms to 300ms.
- Skeleton shimmer may animate continuously.
- Sidebar open/close and hover transitions may remain, but must not delay core task completion.
- No dramatic motion during auth restore, toast display, or retry flows.

## Screen-Level Contracts

### Dashboard
- Keep existing card-grid mental model.
- Replace local fetching state with React Query-backed content states.
- Loading: grid-matched skeleton.
- First-load error: centered compact error card with retry button.
- Empty enrollment state: friendly card with CTA to `/explore`.
- Greeting may use display style, but all supporting metadata stays within label/body roles.

### Explore
- Learning-path listing is part of critical React Query migration.
- Enroll action must show clear pending feedback on clicked CTA, then invalidate related queries.
- Empty/filter-none state uses the shared empty-state card pattern.
- Errors remain toast-first; page-level fallback only when no results can render.

### Lesson
- Lesson content and progress-related UI must preserve the existing lesson layout.
- Loading: lesson-shell skeleton, not spinner.
- Completion-related actions must invalidate dashboard and progress queries.
- Current lesson context should remain visually obvious but not more prominent than the lesson title/content.

### LessonSidebar
- Preserve `?path=` route dependency for this phase.
- Loading uses row skeletons inside sidebar panel.
- Replace English `optional` badge with `Tùy chọn`.
- Replace plain loading/error paragraphs with standardized Vietnamese copy.
- Current lesson row uses active/current state treatment tied to accent rules.

### Quiz
- Use shared loading/error conventions.
- Keep quiz progression visually stable; no full-page spinner takeover.
- Invalidation after submission is conditional to real visible progress dependencies, but lesson completion invalidation remains mandatory.

### Settings
- Migrate visible copy to `vi.ts`.
- Keep settings sections on secondary surfaces with standard 24px padding.
- Use destructive styling only for truly destructive account actions.

### Login/AuthCallback
- Login copy must be fully Vietnamese with proper diacritics.
- Auth callback loading may remain minimal, but if visually exposed for noticeable duration, use a compact skeleton/status surface instead of a generic spinner.

### Onboarding
- Even if not the main migration target, any touched user-facing copy must respect `vi.ts`, typography, color, and skeleton rules.
